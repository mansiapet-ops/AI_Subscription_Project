What Is This Project?
Explain it simply to anyone — technical or non-technical

🎯 The One-Line Explanation

We built an AI-powered banking assistant that automatically finds all your subscriptions (like Netflix, Spotify, gym), predicts when the next payment will be taken, tells you if you're at risk of a payment failing, and generates a smart alert so you can act before money runs out.

🏦 The Real-World Problem (say this to anyone)

Imagine you have a bank account. You have subscriptions to Netflix, Spotify, a gym, electricity bill — maybe 5 to 10 recurring payments. The problem is:

  • You forget which subscriptions are active
  • You don't know when the next payment will be deducted
  • Sometimes your balance is low and the auto-debit fails — the bank charges a penalty
  • Nobody warns you in advance

Our system solves all four problems automatically using AI.

🔢 What we did in simple steps

Step 1

We created fake bank data — 137,806 transactions for 1,200 imaginary Indian customers using a tool called Faker. It generated realistic names, amounts, merchants, and dates.

Step 2

We taught the AI to read transaction descriptions — "NETFLIX.COM MONTHLY SUB" → yes, this is a subscription. "ATM CASH WITHDRAWAL" → no, this is not. This is the NLP (Natural Language Processing) part.

Step 3

We found recurring patterns — if Netflix deducted money on the 14th of every month for 3+ months, that's a confirmed pattern.

Step 4

We predicted the next payment — using ARIMA (a time series model) and Linear Regression, we predict when and how much the next debit will be.

Step 5

We scored the risk — if your balance is low and the payment is large, that's High Risk. We used Gradient Boosting (a machine learning model) for this.

Step 6

We generated alerts — "Upcoming Debit: Amazon ₹499. Risk: Medium. Suggestion: Top up ₹300 before 3 days." These are our GenAI-style alerts.

Step 7

We built a dashboard — a web app using Streamlit where a bank employee can log in and see all this information in charts, tables, and cards.

Pipeline & Code — Explained Simply
Each FR module in plain English + what the code actually does
FR1 · fr1_dataset_generator.py
Synthetic Data Generator Creates fake bank data
Think of it as a "fake customer machine." We use Python's Faker library (Indian locale) to generate 1,200 customers with Indian names, 19 types of subscriptions (Netflix ₹649, Spotify ₹119…), and 137,806 total transactions over 18 months (Jan 2023 – Jun 2024).
FR2 · fr2_data_cleaning.py
Data Cleaning Makes messy data usable
Converts text to lowercase, removes special characters, fills in missing values (null balance → 0), removes duplicate rows. Output: transactions_cleaned.csv
FR3 · fr3_nlp_detector.py
NLP Subscription Detection The AI reading the descriptions
How it works: Reads the Description column (e.g. "NETFLIX.COM MONTHLY SUB"), converts text to numbers using TF-IDF, then Logistic Regression classifies it as subscription (1) or not (0). spaCy helps lemmatize (Netflix → netflix). Accuracy: 99.81%.
FR4 · fr4_pattern_detector.py
Pattern Detection Finds repeating transactions
Groups transactions by Customer + Merchant. If a debit appeared 3 or more times, it's flagged as a recurring pattern. Detects whether it's Monthly (~30 days gap) or Weekly (~7 days). Found 74,582 confirmed patterns.
FR5 · fr5_prediction_engine.py
Debit Prediction When will next payment be?
Date: Linear Regression on past dates (70%) blended with median gap (30%). Amount: ARIMA(1,0,0) — AR(1) means "use the previous value to predict the next." We use order (1,0,0) because subscription amounts are stable (stationary). 3,067 of 3,068 predictions used ARIMA.
FR6 · fr6_risk_scoring.py
Risk Scoring Will the payment fail?
Gradient Boosting uses 8 features (balance, fail rate, upcoming amount…) to output a score 0–1. Score ≥ 0.65 → High Risk, 0.35–0.65 → Medium, below 0.35 → Low. We fixed a "score collapse bug" by using 95th-percentile normalisation instead of fixed thresholds.
FR7 · fr7_genai_alerts.py
GenAI Alert Generator Smart messages for users
Generates structured alert messages like: "Upcoming Debit: Netflix ₹649 | Risk: High | Suggestion: Top up ₹300 before 2 days." Uses Phi-2 style structured templates. Generated 1,200 alerts (one per customer).
FR8 · fr8_insights.py
Insights Generation Summary for each customer
Generates a plain English insight: "You have 2 active subscriptions. Total monthly spend: ₹4,286. Next debit: 2024-07-07." Simple aggregation + text template filling.
FR9 · dashboard.py
Streamlit Dashboard The visual web interface
A web app that runs with streamlit run dashboard.py. Has login, sidebar navigation, 6 tabs: Active Subscriptions, Upcoming Debits, Risk Alerts, Monthly Spend chart, GenAI Alerts, Add New Customer.

# Each FR reads the previous FR's output file FR1 → transactions_raw.csv FR2 → transactions_cleaned.csv FR3 → transactions_nlp.csv (+ saves nlp_model.pkl) FR4 → transactions_patterns.csv + recurring_summary.csv FR5 → predictions.csv (+ saves prediction_model.pkl) FR6 → risk_scored.csv (+ saves risk_model.pkl) FR7 → alerts.txt FR8 → customer_insights.csv FR9 → reads ALL files → shows in dashboard

🏦 Project Overview
Q
What is this project about in simple terms?
▾
We built an AI-powered banking system that automatically detects subscription transactions in bank statements, predicts when the next auto-debit will happen, scores the risk of payment failure, and generates smart alerts. Think of it as a smart assistant that watches your bank account and warns you before your Netflix payment fails because of low balance.
Q
What problem does this solve?
▾
Four key problems: (1) Users forget active subscriptions — money leaks silently. (2) No advance warning before auto-debit. (3) Auto-debits fail due to insufficient balance, causing penalty charges. (4) Subscription transactions are buried in long transaction histories — hard to find manually. Our system addresses all four automatically using NLP, ML, and GenAI.
Q
Who uses this system?
▾
Two personas from the BRD: (1) Retail Customer — uses multiple subscriptions, wants alerts before deductions. (2) Banking Analyst — monitors auto-debit success rates, uses the dashboard to identify at-risk customers and improve retention. The Streamlit dashboard is built for bank employees to monitor customer risk.
Q
What are the 9 Functional Requirements (FRs) from the BRD?
▾
  • FR1 — Dataset Input: Accept/generate transaction dataset with required fields
  • FR2 — Data Processing: Clean, normalize text, handle missing values
  • FR3 — NLP Subscription Detection: Identify subscription transactions using text classification
  • FR4 — Recurring Pattern Detection: Detect frequency and recurring dates
  • FR5 — Debit Prediction: Predict next debit date and amount
  • FR6 — Risk Prediction: Predict failure risk with score
  • FR7 — GenAI Alert Generator: Generate proactive alerts
  • FR8 — Insights Generation: Summarise subscription behaviour
  • FR9 — Dashboard: Display everything in an interactive UI
All 9 FRs are implemented and verified. ✅
📊 Data & FR1–FR2
Q
Why did you use synthetic data instead of real bank data?
▾
Privacy and security reasons. Real bank transaction data contains sensitive personal and financial information — we cannot legally access or use it for a student project. Synthetic data (generated with Faker) allows us to simulate realistic banking behaviour without violating any privacy regulations. The BRD also explicitly states synthetic data as the approach (Section 12). The data is realistic enough to demonstrate and evaluate the system.
Q
How did you generate the dataset? What is Faker?
▾
Faker is a Python library that generates fake but realistic data. We used the en_IN (Indian English) locale to generate Indian-sounding names like "Karan Verma," "Priya Nair." The script creates 1,200 customers, each with: a monthly salary credit, random subscriptions from a catalogue of 19 services (Netflix, Spotify, gym, electricity, etc.), random non-subscription debits (groceries, ATM, UPI), and simulated failed payments. Random noise like occasional null values and small amount variations were added to make it realistic.
Q
How many transactions and what fields does the dataset have?
▾
137,806 transactions for 1,200 customers over 18 months (Jan 2023 – Jun 2024). Fields: CustomerID, CustomerName, TransactionID, Date, Description, Amount, Balance, Merchant, TransactionType (CREDIT/DEBIT), SubscriptionFlag (0 or 1), Status (SUCCESS/FAILED), Frequency (monthly/weekly). This matches the BRD's required fields plus additional engineered columns.
Q
What data cleaning was done in FR2?
▾
FR2 performs: (1) Null handling — missing Balance values filled with 0 or forward-fill. (2) Text normalisation — descriptions converted to lowercase, special characters removed, extra spaces stripped. (3) Duplicate removal — identical transaction rows dropped. (4) Type conversion — Date column parsed to datetime format. Output is transactions_cleaned.csv which all downstream modules read.
🧠 NLP — FR3
Q
What is NLP and how did you use it here?
▾
NLP = Natural Language Processing — teaching a computer to understand human text. Here, we use it to read transaction descriptions like "NETFLIX.COM MONTHLY SUB" and classify them as subscription (1) or not (0). The process: (1) Clean text → (2) TF-IDF converts words to numbers → (3) Logistic Regression predicts class → (4) False-positive filter removes salary/refunds. spaCy is used for lemmatization (e.g., "subscriptions" → "subscription") before vectorization.
Q
What is TF-IDF and why did you use it?
▾
TF-IDF = Term Frequency - Inverse Document Frequency. It converts text into numbers that a machine learning model can understand. TF = how often a word appears in a transaction. IDF = how rare that word is across all transactions. A word like "subscription" appearing in few rows gets a high IDF score — it's informative. A word like "the" appearing everywhere gets a low score — it's not useful. We used n-grams (1–3 words) so phrases like "MONTHLY SUB" are treated as a single feature. Max features: 8,000. sublinear_tf=True reduces the impact of very frequent words.
Q
Why Logistic Regression for classification?
▾
Logistic Regression is ideal here because: (1) Binary classification — we only need yes/no (subscription or not). (2) Fast and interpretable — easy to explain which features (words) influence the decision. (3) Works well with TF-IDF sparse matrices — the high-dimensional, sparse feature space is handled efficiently. (4) class_weight="balanced" handles the imbalance (not all transactions are subscriptions). Alternative considered: SVM and Neural Networks — but Logistic Regression gave 99.81% accuracy, which exceeds the BRD target of >90%, so there was no need for more complexity.
Q
What is the false-positive filter and why is it important?
▾
A false positive is when the model incorrectly identifies something as a subscription when it isn't. For example, "MONTHLY SALARY CREDIT" contains the word "monthly" — the TF-IDF model might flag it as a subscription. To fix this, we maintain a block-list of false-positive keywords: {salary, refund, neft, imps, rtgs, dividend, interest, bonus, deposit}. If any of these words appear in the description, we override the model's prediction to non-subscription, regardless of the model's confidence. This significantly reduced false positives without reducing recall.
Q
How did you achieve 99.81% accuracy?
▾
Several design choices combined: (1) Clean, structured descriptions in our synthetic dataset — real-world bank data uses relatively consistent description formats. (2) Good feature engineering — 1–3 n-grams capture multi-word patterns like "MONTHLY SUB," "ANNUAL PLAN." (3) False-positive filter eliminates clear non-subscriptions. (4) Balanced class weights prevent bias toward the majority class. The 99.81% is on test set (20% of data, stratified split). BRD target was >90% — we exceeded it significantly.
Q
What role does spaCy play?
▾
spaCy (en_core_web_sm model) performs lemmatization — converting words to their base form. For example: "subscriptions" → "subscription," "billed" → "bill," "renewing" → "renew." This reduces vocabulary size and helps the TF-IDF vectorizer generalize better. spaCy is used as an optional preprocessing step — if it's not installed, the pipeline still runs with basic regex cleaning. It also handles tokenization and stop-word removal.
📈 Pattern Detection & Prediction — FR4, FR5
Q
How does recurring pattern detection work?
▾
FR4 groups the NLP-tagged transactions by CustomerID + Merchant. For each group, it: (1) Counts the occurrences — must be ≥ 3 to be considered recurring. (2) Calculates the average gap between consecutive payments. (3) Classifies frequency: gap ≤ 9 days → Weekly, ≤ 35 days → Monthly, otherwise → Irregular. (4) Outputs the first and last seen dates, average amount, and frequency label. Result: 74,582 confirmed recurring patterns from 75,428 NLP-detected subscriptions.
Q
What is ARIMA and why did you choose it for amount prediction?
▾
ARIMA = AutoRegressive Integrated Moving Average. It's a time series forecasting model. The three parameters (p,d,q) mean: p=1 (use 1 previous value to predict the next — AR part), d=0 (no differencing — data is already stationary), q=0 (no moving average terms). We chose ARIMA(1,0,0) because subscription amounts are stationary — they stay roughly the same month to month (Netflix is always ~₹649). We initially used ARIMA(1,1,1) which applied differencing — this was wrong because it changed stable data unnecessarily and degraded prediction quality. After correcting to (1,0,0), 99.97% of predictions used ARIMA successfully.
Q
How do you predict the next debit date?
▾
We use a blend of two methods: (1) Linear Regression (70%) — fits a line through historical debit dates (converted to ordinal numbers) and extrapolates the next date. (2) Median Gap (30%) — takes the median of all historical gaps between consecutive debits (e.g., consistently 30 days). The final predicted date = 0.7 × LR prediction + 0.3 × last_date + median_gap. The blending makes predictions more robust — LR captures trend while median gap handles regularity.
Q
Why not use Prophet or a Neural Network for prediction?
▾
Prophet (by Facebook) is powerful but overkill for subscription data — it's designed for complex seasonal patterns with hundreds of data points. Our subscription data has very regular, short time series (only 10–15 data points per subscription). Neural networks need large amounts of data to train effectively — with only ~15 data points per subscription, they would overfit. ARIMA(1,0,0) is perfect here — simple, fast, interpretable, and works well with short stationary series. The BRD mentions "Time Series" and "Regression" which we both implemented.
⚠️ Risk Scoring — FR6
Q
What is Gradient Boosting and why did you use it for risk scoring?
▾
Gradient Boosting is an ensemble ML algorithm — it builds many decision trees sequentially, each one learning from the mistakes of the previous. It's excellent for tabular data (like our feature table) because it handles non-linear relationships, doesn't need feature scaling, and is robust to outliers. We chose it over Logistic Regression (which we used for NLP) because risk scoring involves more complex interactions — for example, a customer with low balance AND high upcoming debit is much more risky than either factor alone. Gradient Boosting captures these interactions naturally.
Q
What features did you use for risk scoring?
▾
8 features: (1) Current_Balance — how much money the customer has. (2) Failed_Debit_Rate — % of past debits that failed. (3) Avg_Debit_Amount — average past debit size. (4) Upcoming_Total_Debit — total amount predicted to be debited soon. (5) Balance_To_Debit_Ratio — balance / upcoming debit. (6) Upcoming_Debit_Pct — upcoming debit as % of balance. (7) Total_Monthly_Sub_Amount — total monthly subscription burden. (8) Subscription_Count — number of active subscriptions.
Q
What was the risk score collapse bug and how did you fix it?
▾
The bug: Gradient Boosting's predict_proba() was outputting near-certain values — almost all scores were either 0.001 or 0.999. This meant most customers were classified as either Low (346) or High (259), with very few Medium — the distribution was bimodal and unrealistic.

Root cause: The original composite score formula used hard-coded absolute thresholds (e.g., "if balance < ₹5000, score += 0.5"). These thresholds didn't scale across customers with very different income levels.

Fix: Replaced fixed thresholds with 95th-percentile dataset-relative normalisation. For each feature, we compute the 95th percentile of that feature across all customers, then normalise: score = value / p95_value, clipped to [0,1]. This makes risk scores relative to the dataset population, not absolute values. Result: a realistic distribution — 595 Medium, 346 Low, 259 High.
Q
How are risk levels determined?
▾
The composite risk score (0–1) is calculated as a weighted sum of normalised features: 35% failed debit rate + 30% subscription count + 20% upcoming debit amount + 15% subscription burden. Then: Score ≥ 0.65 → High Risk (259 customers), Score 0.35–0.65 → Medium Risk (595 customers), Score < 0.35 → Low Risk (346 customers). The weights reflect business priority — a history of failed debits is the strongest indicator of future failure.
🤖 GenAI & Insights — FR7, FR8
Q
What is GenAI and how did you implement it?
▾
GenAI = Generative Artificial Intelligence — AI that generates new content (text, images, etc.). For our system, the GenAI component (FR7) generates personalised alert messages. The BRD specifies Microsoft Phi-2 or Ollama as options. We implemented a structured template-based approach (Phi-2 style) because: (1) Phi-2 requires significant GPU memory, which may not be available in all environments. (2) The structured approach produces consistent, reliable alerts without hallucination risk. (3) The BRD allows "rule-based fallback." Each alert contains: merchant name, amount, risk level, timing, balance shortfall, and a specific actionable suggestion.
Q
What does a typical alert look like?
▾
CUSTOMER: CUST100592 [FR7] ALERT Upcoming Debit: Amazon ₹499 | Risk: Medium Suggestion: Maintain ₹548 before 2024-07-07. [FR6] RISK ASSESSMENT Score: 0.533 | Level: Medium Reason: Upcoming debit ₹4,270 is high; Monthly sub burden ₹77,162 (6.8% of balance) Balance: ₹1,126,975 | Upcoming: ₹4,269 [FR8] INSIGHT 2 active subscriptions | Monthly spend: ₹4,286
Q
What does FR8 do?
▾
FR8 (Insights Generation) aggregates per-customer data and generates a plain English summary: "You have 2 active subscription(s). Total monthly spend: ₹4,286.77. Next debit: 2024-07-07." It reads from predictions.csv, risk_scored.csv, and calculates: number of active subscriptions (monthly vs weekly), total monthly spend, upcoming debit total and date, current balance, and risk level. Output is customer_insights.csv — each row has a FR8_Message column with the human-readable insight.
🖥️ Dashboard — FR9
Q
Why Streamlit? Why not Flask or Django?
▾
Streamlit is ideal for ML/data science dashboards because: (1) No frontend knowledge needed — pure Python, no HTML/CSS/JS required. (2) Rapid development — a working dashboard in hours, not days. (3) Built-in chart support — direct integration with Plotly, Matplotlib. (4) Reactive by default — UI automatically re-renders when user interacts. Flask/Django would require separate frontend development (React or vanilla JS), which is beyond the project scope. The BRD specifies Streamlit specifically.
Q
Why Plotly instead of Matplotlib for charts?
▾
Plotly produces interactive charts — users can hover to see values, zoom in, filter, and download the chart as PNG. Matplotlib produces static images. For a banking dashboard where the user needs to explore data (hover over a bar to see exact spend), interactive charts provide much better UX. Plotly also renders natively in Streamlit with st.plotly_chart().
Q
How does the login work?
▾
Streamlit's st.session_state is a dictionary that persists across reruns of the script. We store st.session_state["logged_in"] = True after successful login. On each page load, the script checks this flag — if False, show login form; if True, show the dashboard. Credentials are stored in a Python dictionary in the code: USERS = {"banker01": "pass123", "admin": "admin123"}. In production, this would connect to a secure authentication service.
Q
What was the dashboard issue you fixed?
▾
Two issues fixed: (1) Deprecation warning — newer versions of Streamlit deprecated use_container_width=True parameter. We migrated to st.container(border=True) for card-style containers and removed the parameter from places that triggered the warning. (2) Code bloat — the original dashboard was ~1,800 lines because the old version's code was kept as comments at the bottom. We removed all dead code, reducing the file to ~370 lines without changing any functionality. This makes the code much easier to read, explain, and maintain.
🔬 Technical Deep Dives
Q
What is a machine learning pipeline and how does yours work?
▾
A pipeline is a sequence of processing steps where each step's output becomes the next step's input. Our pipeline has 9 steps (FR1–FR9). Data flows as CSV files between modules — each module reads the previous module's output file and writes its own. Trained models are saved as .pkl files using joblib so they don't need to be retrained every time the dashboard is used. The pipeline is orchestrated by run_pipeline.py which calls each module in sequence. Total runtime: 126 seconds for 137,806 transactions.
Q
What does joblib do and why do you save .pkl files?
▾
joblib serialises (saves) Python objects to disk in binary format. We save the trained models (nlp_model.pkl, risk_model.pkl, etc.) so that the dashboard can load them instantly without retraining. Training takes 2–5 minutes; inference (using a saved model) takes milliseconds. The dashboard's "Add New Customer" tab loads subscription_model.pkl to classify new transactions in real-time — this is only possible because we saved the trained model. The .pkl format = Python pickling = serialised binary representation of the object.
Q
Why did you use a sklearn Pipeline for NLP?
▾
Scikit-learn's Pipeline chains preprocessing steps with the final model into a single object: Pipeline([("tfidf", TfidfVectorizer(...)), ("clf", LogisticRegression(...))]). Benefits: (1) No data leakage — TF-IDF is fit only on training data, not test data. (2) Single object to save — joblib.dump(pipeline, "nlp_model.pkl") saves both the vectorizer AND the classifier together. (3) Simple inference — pipeline.predict(new_text) automatically vectorizes then classifies. Without Pipeline, you'd have to save two separate objects and remember to apply them in order.
Q
How would this work with real bank data?
▾
With real data, we would: (1) Connect to a banking API (Plaid, RazorpayX, or internal bank API) instead of generating synthetic data. (2) Replace FR1's Faker generator with an API data ingestion module. (3) Keep FR2–FR9 mostly unchanged — the pipeline is data-agnostic. (4) Add data encryption for PII (personally identifiable information). (5) Deploy on a cloud server (AWS/GCP) instead of running locally. (6) Add real authentication (OAuth2, bank's SSO). The BRD explicitly marks real-time bank integration as "Out of Scope" for this project.
Q
What are the non-functional requirements and how did you meet them?
▾
From BRD Section 10: (1) Performance <10 sec processing — dashboard loads data using @st.cache_data so after first load, all subsequent loads are instant. Pipeline itself takes 126 seconds for full 137K dataset, but individual customer queries are <1 sec. (2) Local Execution Required — entire system runs with python run_pipeline.py and streamlit run dashboard.py. No cloud services or internet required (except initial package install). (3) Simple UI — sidebar navigation, clear tabs, colour-coded risk cards. (4) High Accuracy — NLP 99.81% vs target 90%.
🔧 Challenges & Decisions
Q
What was the hardest challenge you faced?
▾
The risk score collapse bug in FR6 was the most technically challenging. The Gradient Boosting model's scores were collapsing to near-0 or near-1 for all customers, giving an unrealistic binary distribution. Debugging required understanding that the issue was in the composite scoring formula, not the ML model itself — fixed thresholds didn't account for different customer income scales. The 95th-percentile normalisation solution was non-obvious and required several iterations to arrive at. The second hardest was the ARIMA order selection — understanding why (1,1,1) was wrong for stationary data required revisiting time series stationarity theory.
Q
What would you do differently if you had more time?
▾
(1) Real LLM integration — actually call Phi-2 or Llama for richer, more natural alert text instead of template-based. (2) Real bank API — connect to Plaid or a sandbox banking API for live data. (3) Push notifications — integrate Twilio/MSG91 to send actual SMS alerts. (4) Anomaly detection — use Isolation Forest to flag one-time charges that look like subscriptions. (5) Model evaluation — add proper cross-validation and metrics dashboard for FR5 (MAE, RMSE for prediction accuracy). (6) User-facing mobile app — the dashboard is for bankers; a mobile app for customers would complete the vision.

🗣️ One-liner explanations

TF-IDF "Converts transaction descriptions into numbers that the AI can understand — rare words like 'subscription' get high scores."
ARIMA(1,0,0) "A time series model that uses the previous payment amount to predict the next one. We use (1,0,0) because subscription amounts don't change much over time."
Gradient Boosting "An ML model that builds many decision trees and combines them — great for predicting risk from multiple features like balance, failed payments, and subscription count."
spaCy "Preprocessing tool that converts 'subscriptions' to 'subscription' — reduces vocabulary size and improves NLP accuracy."
Streamlit "A Python library that lets data scientists build web dashboards without knowing HTML or JavaScript."
95th percentile normalisation "Instead of using fixed thresholds, we compare each customer's risk factors relative to the top 5% of customers — makes risk scores meaningful across different income levels."
Pipeline (sklearn) "Chains TF-IDF and Logistic Regression into one object — prevents data leakage and makes saving/loading the model trivial."
Stationary data "Data where the mean and variance don't change over time. Netflix is always ~₹649 — that's stationary. So we use ARIMA(1,0,0) with d=0 (no differencing needed)."
